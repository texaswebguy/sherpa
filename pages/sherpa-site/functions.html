Rendering Text

Rendering data from view model

In the old days of HTML we just slapped content inside html tags and ran with it. With Sherpa we want to stress the importance of building a prototype from data.

Hello, !

Markup:

<h2>Hello, <span data-bind="text: $root.user.name()"> </span>!</h2>

How it works:

user.name:
user is an element in the "viewModel". It is at the top level data set on the page available to Sherpa.

if you look in assets/js/app.js you will see the following:
addToViewModel = {
	user: {
		name:ko.observable("Carl Weatherspoon")
	},
    question: ko.observable('How many roads must a man walk down before you can call him a man?')
};

Try changing the name property, save and reload.
We'll explain "observable after this..."

Reactive Text

In the example above, you created a text from data. What you might have not known is that the data is wired to the page because of ko.observable

User Name:

Hello, !

Markup:

<p>User Name: <input data-bind="value: $root.user.name()" /></p>
<h2>Hello, <span data-bind="text: $root.user.name()"> </span>!</h2>

How it works:

Above you might have noticed that instead of name:"Carl Weatherspoon" we used name:ko.observable("Carl Weatherspoon").

ko.observable wires the data to the page and anytime it changes the rendering on the page refreshes without the page reloading.

Rendering a textkey

Prototypes have to be tested around the globe. One of the most important things is to externalize all text so that content is easily translated. For this, Sherpa provide the "msg" tag.

Markup:

<h2 data-bind="msg: 'sample_greeting'"></h2>

How it works:

msg is what is called a custom binding for knockout.js which is one of the foundations of the Sherpa framework.
The msg tag takes 'sample_greeting' and looks for it in the viewModel.

You can right click this page, click on "inspect element" and then click on "console"
Now type > viewModel.content.sample_greeting <
You will see the text rendered above.

Now, for real fun go to functions.html?l=cn and watch your text change.

Don't forget to go back to english.

You can also pre-process a textkey in your app dataset (viewModel): "pre_processed_text" : Sherpa.msg('sample_greeting');
<span data-bind="text: 'pre_processed_text'"></span>

Truncating text

Sometimes there is a need to truncate text that comes from a data source. Optionaly you might have a need for the truncated text to show a more/less link.

Lorem ipsum dolor sit amet, consectetur adipisicing elit, sed do eiusmod tempor incididunt ut labore et dolore magna aliqua. Ut enim ad minim veniam, quis nostrud exercitation ullamco laboris nisi ut aliquip ex ea commodo consequat. Duis aute irure dolor in reprehenderit in voluptate velit esse cillum dolore eu fugiat nulla pariatur. Excepteur sint occaecat cupidatat non proident, sunt in culpa qui officia deserunt mollit anim id est laborum.

Markup:

<p data-bind="truncate: {length:50,showMore:true}"> ... </p>

How it works:

The configuration object is used to determine the length by characters of truncation and several options. It is possible to use the function with no options which then the system will default to 100 characters.

Examples of possible options:

<p data-bind="truncate: {}"> ... </p>
<p data-bind="truncate: {length:50,showMore:true}"> ... </p>
<p data-bind="truncate: {
	length:50,
	showMore:true, 
	moreText:'show more', 
	lessText:'show less', 
	elipse:'...'
}"> ... </p>

If not specified, moreText, lessText, and elipse are looked for in the messages file (assets/content/messages_en.json) under text_truncate_more, text_truncate_less and text_truncate_elipse.

Dummy text

Sometimes there is a need to render dummy text. Dummy text can be rendered by number of characters or number of paragraphs.

Markup:

<p data-bind="loremipsum: 200"> ... </p>

How it works:

There is an array of pre-loaded dummy text in Sherpa found in viewModel.core_config.lorem_ispum. It is possible to use the function with options to provide paragraphs of dummy text.

Examples of possible options:

<p data-bind="loremipsum: 200"> ... </p>
<div data-bind="loremipsum: {paragraphs:5}"> ... </div>
<div data-bind="loremipsum: {paragraphs:5,length:300}"> ... </div>
<p data-bind="loremipsum: {startLorem:true,length:300}"> ... </p>

startLorem will assure that the dummy text starts with "Lorem ispum dolor sit amet "
length refers to the length in characters in each paragraphs.
Notice the use of div whenever paragraphs option is present.  This is because Sherpa will return dummy text inside p tags.

Converting Currency

Prototypes have to be tested around the globe. Realistic rendering of price data is vital to the success of a test. For this, Sherpa provides the "formatcurrency" tag.

Markup:

<h2 data-bind="formatcurrency: {amount:499.00,format:'decimal'}"></h2>

How it works:

formatcurrency is what is called a custom binding for knockout.js which is one of the foundations of the Sherpa framework.
The formatcurrency tag takes '499.00' or any data referencing a number and converts is in accordance to the configuration for each locale.

You can right click this page, click on "inspect element" and then click on "console"
Now type > $root.config.currency <
You will see the currency configuration object.

Now, for real fun go to functions.html?l=cn and watch your currency change.

Dynamic Display

Sometimes in a prototypes you need a way to display things conditionaly. There are couple of ways to do this: - Knockout JS if/ifnot ko comments using boolean variable or conditional function - Knockout JS if/ifnot bindings using boolean variable or conditional function

I display beacuse $root.config.some_boolean_variable is set to true

I display beacuse $root.config.some_other_boolean_variable is set to false

I display because ?boolean_example=true is not in the url. Add it to the url and watch me disappear!

I display because you added ?boolean_example=true to the url!

I display beacuse $root.config.some_boolean_variable is set to true (through ko comments)

I display beacuse $root.config.some_other_boolean_variable is set to false (through ko comments)

Markup:

<p data-bind="if: $root.config.some_boolean_variable" class="purple-text">I display beacuse $root.config.some_boolean_variable is set to true</p>

<p data-bind="ifnot: $root.config.some_other_boolean_variable" class="green-text">I display beacuse $root.config.some_other_boolean_variable is set to false</p>

<p data-bind="ifnot: $root.some_boolean_function()" class="blue-text">I display because ?boolean_example=true is not in the url.  Add it to the url and watch me disappear!</p>

<p data-bind="if: $root.some_boolean_function()" class="orange-text">I display because you added ?boolean_example=true to the url!</p>

<!-- ko if: $root.config.some_boolean_variable -->
<p class="berry-text">I display beacuse $root.config.some_boolean_variable is set to true (through ko comments)</p>
<!-- /ko -->

<!-- ko ifnot: $root.config.some_other_boolean_variable -->
<p class="red-text">I display beacuse $root.config.some_other_boolean_variable is set to false (through ko comments)</p>
<!-- /ko -->

How it works:

Conditionals will remove the html when the condition fails. In cases where you want to hide the html but use it later the visible binding should be used.

Take a look at assets/js/cookbook.js and you will see how all the conditionals used above are set up:


addToConfigs = {
	sample_custom_config : true,
    some_boolean_variable : true,
    some_other_boolean_variable : false,
}

var some_boolean_function = function(){
	var q = Sherpa.urlQuery();
	if(q.boolean_example === "true") {
		return true;
	} else {
		return false;
	}
}

Reactive Show / Hide

Show and hide is like dynamic display but it differs that whatever you bind with "visible" is still in the document and it is available to display later. Take for example a checkbox that reveals something when it is checked:

I'm visible because the checkbox is checked


Hi! I'm alpha, beta and charlie are hiding right now!

Hi! I'm beta, alpha and charlie are hiding right now!

Hi! I'm charlie, alpha and beta are hiding right now!

Markup:

<label><input type="checkbox" data-bind="checked: $root.checkbox_value" /> Check me to see something</label>
<p data-bind="visible: $root.checkbox_value()">I'm visible because the checkbox is checked</p>

<label>Select something: </label>
<select data-bind="options: $root.dropdown_values, value: $root.selected_dropdown_value"></select>
<p data-bind="visible: $root.selected_dropdown_value() === 'alpha'">Hi! I'm <strong>alpha</strong>, beta and charlie are hiding right now!</p>
<p data-bind="visible: $root.selected_dropdown_value() === 'beta'">Hi! I'm <strong>beta</strong>, alpha and charlie are hiding right now!</p>
<p data-bind="visible: $root.selected_dropdown_value() === 'charlie'">Hi! I'm <strong>charlie</strong>, alpha and beta are hiding right now!</p>

How it works:

Typically you see this sort of action built in a jquery show, hide, or toggle functions.

The difference here is that this is built using reactive data.  When the data changes, the UI responds based on the condition set during the initial rendering.

In our viewmodel we have set checkbox_value to ko.observable(false).

This sets the initial sate of our checkbox to unchecked: <input type="checkbox" data-bind="checked: $root.checkbox_value" />

Our show/hide text is bound with data-bind="visible: $root.checkbox_value()" - notice the () after checkbox_value. This fires to the observable function and now the visible condition is wired to the data and any time it validates as "true" it will re-render what it is inside the binding.

Same thing is happening with the dropdown menu example: $root.selected_dropdown_value() is an observable that is being actively changed by the dropdown selection.

You should notice that once wired, the reactive nature of the rendering is much more natural than building custom jquery functions to explicitly call out elements to show and hide.

Reactive Fade

Reactive Fade is like show / hide. only difference is that the elements fade in and out rater than just show and hide. There are two options: visibleFade and visibleFadeToggle. visibleFade is suitable for actions that will affect only one element. For multiple elements you will want to use visibleFadeToggle which will immediately hide all other elements rather than fade them and create a jerky motion.

Hi! I'm beta, alpha and charlie are hiding right now!

Hi! I'm charlie, alpha and beta are hiding right now!

<p data-bind="visibleFade: $root.checkbox_fade_value()">I'm visible because the checkbox is checked</p>

<label>Select something: </label>
<select data-bind="options: $root.dropdown_values, value: $root.selected_fade_dropdown_value"></select>
<p data-bind="visibleFadeToggle: $root.selected_fade_dropdown_value() === 'alpha'">Hi! I'm <strong>alpha</strong>, beta and charlie are hiding right now!</p>
<p data-bind="visibleFadeToggle: $root.selected_fade_dropdown_value() === 'beta'">Hi! I'm <strong>beta</strong>, alpha and charlie are hiding right now!</p>

How it works:

This works exactly as show /hide but it adds a fade transition.

Pseudo Authentication

In a prototype you often have to show what a component might look like when a user is logged in vs not logged in. For this reason we have implemented a very simple system that tags the page as authenticated or not. This is used in conjunction with two css classes that can be used on any elements on the page.

  • authenticated-only: Only displays when the prototype is :"authenticated"
  • not-authenticated-only: Only displays when the prototype is NOT :"authenticated"

There are two ways to trigger a page's pseudo authentication:

  • Query string parameter: myPrototypePage.html?authenticated=true or myPrototypePage.html?authenticated=false
  • Javscript: Sherpa.publish("authenticated",true)

Both of these methods will set the page class and viewModel variable to enable dynamic display:

  • HTML Page classes: authenticated or not-authenticated
  • viewModel: $root.isAuthenticated

Examples:

I display beacuse this page IS authenticated

I display beacuse this page IS NOT authenticated

I display beacuse this page IS authenticated

I display beacuse this page IS NOT authenticated

I display beacuse this page IS authenticated

I display beacuse this page IS NOT authenticated

How to trigger:

html query string: log in

html query string: log out

Pseudo login component:

Markup:

Example:

Use to explicictly remove the element not used:						
<p data-bind="if: $root.isAuthenticated">I display beacuse this page IS authenticated</p>
<p data-bind="ifnot: $root.isAuthenticated">I display beacuse this page IS NOT authenticated</p>

Use when you still want to access the other element:
<p class="authenticated-only">I display beacuse this page IS authenticated</p>
<p class="not-authenticated-only">I display beacuse this page IS NOT authenticated</p>

Use when you want to conditionaly refer to different components:
<!-- ko if: $root.isAuthenticated -->
<div data-bind="component: { 
	name :'foobar', 
	type :'authenticated',
}"></div>
<!-- /ko -->
<!-- ko ifnot: $root.isAuthenticated -->
<div data-bind="component: { 
	name :'foobar', 
	type :'unauthenticated',
}"></div>
<!-- /ko -->

Trigger:
<a href="functions.html?authenticated=true#pseudo-authentication">log in</a>
<a href="functions.html?authenticated=false#pseudo-authentication" >log out</a>

<div class="box sherpa-gray-stroke rounded-small" data-bind="component: { 
	name :'forms', 
	type :'pseudo-login',
	options : {
		forgot_password: false,
        create_account: false
	}
}">
</div>

How it works:

The trigger mechanism adds either "authenticated" or "not-authenticated" to the html element.  There is css in Sherpa to work with this to provide classes that automatically:

html.authenticated .authenticated-only, 
html.not-authenticated .not-authenticated-only {
  display:block;
}


html.authenticated .not-authenticated-only, 
html.not-authenticated .authenticated-only {
  display:none;
}

Creating Hotkey Functions

In a prototype you might have a need to create hotkeys for either supporting information regarding the prototype or as part of functionality required in the design itself.

The hotkey functionality is provided by Keith Cirkel's jwerty jQuery plugin.

Specifying hotkey functionality can be done inside javascript. The best place would be your app.js file.

Examples:

Press ctrl-x for demo

Markup:

<p class="btn btn-secondary">Press ctrl-x for demo</p>
<script type="text/javascript">Sherpa.key('ctrl+x', function () { alert("Hello x!") });</script>

How it works:

The code for the button does not do anything functional. Only the javascript is what makes this function.
Complex key combinations are possible, refer to the jwerty link above for details.

Creating a "Page"

The Sherpa Starter Kit comes with a sample project index page.

A project index page has several critical functions:

  • Initiates all Sherpa Prototype Framework functionality
  • Initiates all prototype data
  • Provides access to multiple page templates
  • Provides access to multiple virtual pages
  • Provides access to project information

You can build separate html files for each page in your prototype but using a project index page allows the framework to be much faster.

In the actual code of the project index page you should see two different sections:

  • Index page content
  • Other "pages" in the prototype

Index page content:

Note this line:
<section data-bind="page: {id: 'start', title: 'project name'}">

This section sets up all the links you see in the page when it loads.
id: 'start' designates this section as the default page that will display when the page loads.

Other "pages" in the prototype:

Note this link:
<a href="#!/templates/banner-left-nav">banner-left-nav </a>

This link's href points to an id called "banner-left-nav".
However, you will notice "#!/" being used.  This tells the Sherpa system to look for a "page" with "templates" as id.

Below this you will notice <div data-bind="page: {id: 'templates'}">

This sets up a group of pages that will share the same structure or common elements. You should notice that the first thing inside the template page is <div data-bind="component: {name:'masthead', type:'starter-kit'}"> and at the bottom you should see a call to the footer component.

Both of these will render with any id present inside the template page.

Now look under the masthead and you will see <div data-bind="page: {id: 'banner-left-nav'... which renders when http://localhost:8080/project_index.html#!/templates/banner-left-nav is called.

How it works:

All this code is using a knockout.js library called pager.js.  You can learn more about it at http://pagerjs.com/

Creating a Theme

The Sherpa Starter Kit comes with the default Sherpa css theme.

CSS themes can be built using less and provide control of global elements such as colors, heights, widths, paddings, etc.

In addition, some component level variables are provided to make theme adjustments.

If further customization is provided, a theme can also declarations that override Sherpa classes or a whole new non sherpa css can be specified.

There are three ways to specify or switch a theme:

  • Configuration of default theme
  • In the url
  • Via JavaScript

Configuration of default theme:

By editing assets/config/global_config.js you can change the default theme.

Note this line contains the default them css to be loaded:
{ css_theme: SHERPA.CSS_CORE_PATH+"sherpa.css" }

Switch theme through url:

You might want to provide a link on your project index page that would switch the theme.

For example: See foobar theme. See Sherpa theme.

Via JavaScript:

You can have any function execute this to initiate a page reload with the new theme.

Sherpa.themeSwitcher("foobar");

Special classes for theme logic:

If you create a theme called "foobar" you have two classes available that will provide show/hide logic capability:

  • theme-hide-foobar
  • theme-only-foobar

Note: this uses a display: block; so you can't really use it in an in-line display (word in a middle of a sentence).

Markup:

<div class="theme-hide-foobar">
	This element will only display only when any theme other than "foobar" is in context.
</div>
<div class="theme-only-foobar">
	This element will only display when the "foobar" theme is in context.
</div>
